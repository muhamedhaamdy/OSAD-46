01 Singleton Creational
What it is

The Singleton pattern ensures a class has exactly one instance throughout the entire program, and provides a global access point to that instance. It is not a Singleton if more than one instance can exist.

When to use it
  • A shared resource (database connection pool, logger, config manager, thread pool) must be shared across the app.
  • Creating multiple instances would cause incorrect behaviour or waste resources.
  • You need a global access point without using a global variable.
How it works
  1. Make the constructor private so no external class can call new.
  2. Add a private static field of the same type to hold the one instance.
  3. Add a public static factory method (getInstance()) that lazily creates and returns the instance.
UML Diagram
Java Code Example
public class DatabaseConnection {
    // private static field holds single instance
    private static DatabaseConnection uniqueInstance;

    // private constructor — no external new()
    private DatabaseConnection() {
        System.out.println("Connecting to DB...");
    }

    // public factory method — lazy + thread-safe
    public static synchronized DatabaseConnection getInstance() {
        if (uniqueInstance == null)
            uniqueInstance = new DatabaseConnection();
        return uniqueInstance;
    }

    public void query(String sql) { System.out.println("Executing: " + sql); }
}

// Usage
DatabaseConnection db1 = DatabaseConnection.getInstance();
DatabaseConnection db2 = DatabaseConnection.getInstance();
System.out.println(db1 == db2); // true — same object
Design Principles Applied
  • SRP The class controls its own instantiation — one reason to exist
  • Encapsulation Constructor is hidden; creation logic is private
Interview Questions
Q: Why is the constructor private in a Singleton?
To prevent any external code from calling new ClassName() and creating a second instance. The only way to get an instance is through the controlled getInstance() factory method.
Q: Why add synchronized to getInstance()?
In a multi-threaded environment, two threads could simultaneously find instance == null and both create new instances. synchronized ensures only one thread executes the method at a time, preserving the "one instance" guarantee.
Q: What is the difference between eager and lazy initialization?
Eager: private static Singleton instance = new Singleton(); — created at class load. Lazy: created only when first called. Lazy saves memory; eager avoids thread-safety concerns.
Pitfalls & When NOT to Use
  • Singletons are essentially global state — they make testing hard because you cannot easily swap in a mock.
  • Violates SRP if the Singleton also does too much (manages state AND config AND logging).
  • Not thread-safe by default — always use synchronized or the Bill Pugh holder idiom.
  • Avoid if the concept genuinely needs more than one instance (e.g., multiple DB connections).
02 Factory Creational
What it is

The Factory pattern defines an interface for creating an object, but lets subclasses decide which class to instantiate. The factory method defers instantiation to subclasses. Clients work with the product through the common interface, never calling new directly.

When to use it
  • You don't know ahead of time which class to instantiate — the exact type is determined by subclass or configuration.
  • You want to provide an extension point: allow subclasses or callers to swap the product without modifying the creator.
  • A framework needs to create objects but lets the app decide their concrete types (e.g., different UI buttons per OS).
How it works
  1. Define a Product interface (e.g., Button) with the product's operations.
  2. Create concrete product classes (e.g., WindowsButton, MacButton) that implement it.
  3. Create a Creator class with an abstract createProduct() factory method.
  4. Each ConcreteCreator overrides createProduct() to return its specific product.
  5. Client code uses the Creator; the factory method handles instantiation.
UML Diagram
Java Code Example
// Product interface
interface Button { void render(); }

// Concrete products
class WindowsButton implements Button {
    @Override public void render() { System.out.println("Windows button"); }
}
class MacButton implements Button {
    @Override public void render() { System.out.println("Mac button"); }
}

// Creator — declares the factory method
abstract class Dialog {
    abstract Button createButton(); // factory method
    void render() { createButton().render(); } // uses product
}

// Concrete creators
class WindowsDialog extends Dialog {
    @Override Button createButton() { return new WindowsButton(); }
}
class MacDialog extends Dialog {
    @Override Button createButton() { return new MacButton(); }
}

// Client — picks creator based on OS, never calls new Button()
Dialog dialog = isWindows ? new WindowsDialog() : new MacDialog();
dialog.render(); // correct button rendered automatically
Design Principles Applied
  • OCP Add new products by adding new creator subclasses, not modifying existing code
  • DIP High-level Dialog depends on Button interface, not concrete classes
  • SRP Each creator handles exactly one product type
Interview Questions
Q: What problem does Factory solve that a simple if/else new doesn't?
A direct if/else requires modifying the creator every time a new product is added (violates OCP). Factory externalises creation into subclasses; adding a product means adding a new class with zero change to the client or base creator.
Q: How does Factory differ from Abstract Factory?
Factory creates one product via a single method; the variation is in the subclass. Abstract Factory creates families of related products via multiple factory methods on an interface; it groups related factories together.
Q: Can a Factory Method be static?
Yes — a "static factory method" is a common Java idiom (e.g., LocalDate.of()). However, static methods cannot be overridden, so you lose the polymorphic extension point. The GoF pattern specifically requires an overridable (virtual/abstract) method.
Pitfalls & When NOT to Use
  • Can lead to a class explosion — one creator subclass for every product variant.
  • If only one type will ever exist, the indirection adds unnecessary complexity; a simple constructor or static method is cleaner.
  • Avoid confusing it with Abstract Factory — Factory is a single-method pattern, not a factory object.
03 Prototype Creational
What it is

The Prototype pattern lets you create new objects by cloning an existing object (the prototype) rather than calling a constructor. The clone copies the prototype's state, which can then be tweaked. It delegates the complexity of copying to the object itself.

When to use it
  • Object creation is expensive (e.g., database queries, heavy computation) and an existing similar object can be copied instead.
  • You want to avoid subclassing just to vary the initial configuration of objects.
  • The class to instantiate is specified at runtime (e.g., loaded from config), making new ClassName() impossible at compile time.
  • Examples: cloning document templates, game object spawning, undo/redo snapshots.
How it works
  1. Declare a Cloneable/Prototype interface with a clone() method.
  2. Concrete classes implement clone(), creating a copy of themselves (deep or shallow as needed).
  3. Optionally maintain a Prototype Registry (a map of named prototypes) so clients can look up, clone, and customise without knowing the concrete classes.
UML Diagram
Java Code Example
// Prototype interface
interface Shape {
    Shape clone();
}

// Concrete prototype
class Circle implements Shape {
    int radius;
    String color;

    Circle(int radius, String color) {
        this.radius = radius; this.color = color;
    }

    // Copy constructor — deep clone here
    Circle(Circle source) {
        this.radius = source.radius;
        this.color  = source.color;
    }

    @Override public Shape clone() { return new Circle(this); }
}

// Optional: registry of named prototypes
class ShapeRegistry {
    private Map<String, Shape> registry = new HashMap<>();

    void register(String id, Shape s) { registry.put(id, s); }
    Shape get(String id) { return registry.get(id).clone(); }
}

// Usage
ShapeRegistry reg = new ShapeRegistry();
reg.register("bigRed", new Circle(50, "red"));

Circle c = (Circle) reg.get("bigRed"); // cloned, not new
c.color = "blue"; // tweak without affecting prototype
Design Principles Applied
  • OCP Add new cloneable types without changing client code
  • Encapsulation The object knows how to copy itself — cloning logic stays inside
  • DRY Shared creation logic centralised in clone() rather than scattered constructors
Interview Questions
Q: What is the difference between shallow copy and deep copy in Prototype?
Shallow copy: copies field values — primitive fields are duplicated, object-reference fields point to the same objects as the original. Deep copy: recursively copies all referenced objects so the clone is fully independent. Use deep copy when mutable nested objects must not be shared.
Q: When is Prototype better than using a constructor directly?
When creating an object is expensive (e.g., loading assets, running queries) and you need many similar objects. Cloning a pre-built prototype is typically faster than re-running the expensive initialisation. It also lets you avoid knowing the exact class at call time.
Q: How does Java's built-in Cloneable relate to this pattern?
Java's Object.clone() + Cloneable is a built-in mechanism, but it is shallow by default and CloneNotSupportedException is checked, making it awkward. Most modern code implements a copy constructor or a custom clone() method instead for clarity and safety.
Pitfalls & When NOT to Use
  • Deep copying complex object graphs (circular references, shared state) is error-prone — document clone semantics carefully.
  • Java's Object.clone() is shallow by default and requires casting; prefer copy constructors.
  • Don't use if objects don't have meaningful state to copy — a plain constructor is simpler.

04 Adapter Structural
What it is

The Adapter pattern wraps an existing class (the adaptee) in a new class (the adapter) so that it can be used through a different interface. Like a power plug adapter — same electricity, compatible socket.

When to use it
  • You want to reuse a class that has the right functionality, but the wrong interface.
  • The adaptee is in its own inheritance hierarchy and you can't use multiple inheritance.
  • Integrating a third-party library that you can't modify.
How it works
  1. Define the target interface that the client expects.
  2. Create the Adapter class that implements the target interface.
  3. The Adapter holds an instance of the Adaptee.
  4. Each method in the Adapter translates the call to the Adaptee's method.
UML Diagram
Java Code Example
// Target interface the client knows about
interface MediaPlayer {
    void play(String filename);
}

// Adaptee — has the right logic, wrong method name
class VlcPlayer {
    public void playVlc(String filename) {
        System.out.println("VLC playing: " + filename);
    }
}

// Adapter — bridges the gap
class VlcAdapter implements MediaPlayer {
    private VlcPlayer vlc = new VlcPlayer(); // holds adaptee

    @Override
    public void play(String filename) {
        vlc.playVlc(filename); // translate the call
    }
}

// Client code — only knows about MediaPlayer
MediaPlayer player = new VlcAdapter();
player.play("movie.vlc"); // works transparently
Design Principles Applied
  • Encapsulate What Varies Hides the adaptee's different interface
  • Composition over Inheritance Wraps via has-a, not multiple inheritance
  • OCP Client code doesn't change when adding new adapters
Interview Questions
Q: What is the difference between Adapter and Facade?
Adapter makes two incompatible interfaces work together, usually wrapping a single class. Facade provides a simplified interface over a complex subsystem of many classes — it is about simplification, not compatibility.
Q: When would you use object adapter vs class adapter?
An object adapter (using composition) is preferred in Java since Java doesn't support multiple inheritance. A class adapter (using inheritance) is only possible in languages that support it, like C++.
Pitfalls & When NOT to Use
  • If the adaptee's interface is very different, the adapter can become complex and fragile.
  • Don't use it just to rename methods — only where a genuine interface mismatch exists.
  • Overusing adapters to patch poor design is a code smell.
05 Facade Structural
What it is

The Facade pattern provides a single, simple interface to a complex subsystem of many classes. It doesn't hide the subsystem — it wraps it in a clean, easy-to-use API. Like a hotel concierge who handles all the complexity of booking restaurants, taxis, and tours for you.

When to use it
  • A subsystem has many interdependent classes and is hard to use directly.
  • You want to define a layered architecture and provide an entry point to each layer.
  • You want to decouple client code from implementation details of a complex library.
How it works
  1. Identify the complex subsystem's classes.
  2. Create a Facade class with simple, task-oriented public methods.
  3. The Facade internally coordinates calls to multiple subsystem classes.
  4. Clients call only the Facade — they don't need to know the subsystem exists.
UML Diagram
Java Code Example
// Subsystem classes
class Amplifier  { void on() { System.out.println("Amp on"); } }
class Projector  { void on() { System.out.println("Proj on"); } }
class DVDPlayer  { void play(String m) { System.out.println("Playing: "+m); } }
class Lights     { void dim(int lvl) { System.out.println("Lights @ "+lvl+"%"); } }

// Facade — one method does it all
class HomeTheaterFacade {
    private Amplifier  amp  = new Amplifier();
    private Projector  proj = new Projector();
    private DVDPlayer  dvd  = new DVDPlayer();
    private Lights     lts  = new Lights();

    public void watchMovie(String movie) {
        amp.on(); proj.on(); lts.dim(10); dvd.play(movie);
    }
}

// Client — simple!
new HomeTheaterFacade().watchMovie("Inception");
Design Principles Applied
  • Encapsulate What Varies Hides all subsystem complexity behind one surface
  • Decouple Client Client is shielded from subsystem changes
  • Law of Demeter Client talks to one friend (facade), not all subsystem objects
Interview Questions
Q: Does the Facade prevent access to the subsystem directly?
No. Facade doesn't lock down the subsystem. Clients who need fine-grained control can still access subsystem classes directly. The Facade just provides a more convenient option.
Q: How is Facade different from Adapter?
Adapter changes the interface of an existing object to match what a client expects — it's about compatibility. Facade defines a new, simpler interface over a group of objects — it's about simplification.
Pitfalls & When NOT to Use
  • A Facade can become a "god object" if too much logic is crammed into it.
  • Avoid if the subsystem is already simple — adding a facade is needless complexity.
  • The facade can become a bottleneck if all traffic must pass through it.
06 Proxy Structural
What it is

The Proxy pattern provides a surrogate or placeholder for another object to control access to it. The proxy and the real object share the same interface; the client can't tell the difference. The proxy intercepts calls and can add behaviour (lazy loading, caching, access control, logging) before or after forwarding to the real object.

When to use it
  • Virtual Proxy: Delay expensive object creation until it's actually needed (lazy initialisation).
  • Protection Proxy: Add access control — only authorised clients can use the real object.
  • Remote Proxy: Represent an object in a different address space (e.g., RMI stub is a proxy).
  • Caching Proxy: Cache results of expensive operations and return cached results on subsequent calls.
How it works
  1. Define a Subject interface with the operation(s) both real object and proxy implement.
  2. Create the RealSubject that does the actual work.
  3. Create the Proxy class: it implements Subject, holds a reference to RealSubject (created lazily or injected), and wraps each method with extra logic before/after delegation.
  4. Client uses the Subject interface — it may hold a Proxy or RealSubject interchangeably.
UML Diagram
Java Code Example
// Subject interface
interface Image { void display(); }

// RealSubject — expensive to create
class HighResImage implements Image {
    private String filename;

    HighResImage(String filename) {
        this.filename = filename;
        System.out.println("Loading from disk: " + filename); // expensive!
    }

    @Override
    public void display() { System.out.println("Displaying: " + filename); }
}

// Virtual Proxy — delays loading until display() is first called
class ImageProxy implements Image {
    private String       filename;
    private HighResImage realImage; // null until needed

    ImageProxy(String filename) { this.filename = filename; }

    @Override
    public void display() {
        if (realImage == null)
            realImage = new HighResImage(filename); // lazy load
        realImage.display();
    }
}

// Client — uses Image interface, never knows if it's proxy or real
Image img = new ImageProxy("photo.png");
// photo.png is NOT loaded yet — just a lightweight proxy
img.display(); // NOW it loads and displays
img.display(); // displays immediately — already loaded
Design Principles Applied
  • OCP Add new proxy types (caching, logging) without modifying RealSubject or client
  • SRP Cross-cutting concerns (caching, auth) live in proxy, not in RealSubject
  • Program to Interface Client and proxy both use Subject; concrete types stay hidden
Interview Questions
Q: How is Proxy different from Decorator?
Both wrap an object with the same interface — but intent differs. Proxy controls access to the object (lazy init, auth, remote calls). Decorator adds behaviour to an object dynamically (logging, formatting). Proxy often controls whether to call the real object; Decorator always calls it and adds around it.
Q: What is a dynamic proxy in Java?
Java's java.lang.reflect.Proxy creates proxy instances at runtime without writing a proxy class manually. It intercepts method calls via an InvocationHandler. Widely used in AOP frameworks (Spring), mocking libraries, and ORM systems.
Q: What are the three most common proxy types?
Virtual Proxy (lazy creation/caching), Protection Proxy (access control), and Remote Proxy (local stand-in for a remote object). All share the Subject interface — the distinction is in why they intercept, not how.
Pitfalls & When NOT to Use
  • Response time may increase due to the extra indirection — benchmark for performance-critical paths.
  • Can become complex if the proxy accumulates too many responsibilities (caching + auth + logging = god proxy).
  • Overkill for simple objects where direct access is fine.
07 Decorator Structural
What it is

The Decorator pattern attaches additional responsibilities to an object dynamically by wrapping it inside decorator objects that implement the same interface. Decorators are an alternative to subclassing for extending behaviour — you can stack them arbitrarily.

When to use it
  • You want to add behaviour to individual objects without affecting others of the same class.
  • You need to add/remove responsibilities dynamically at runtime.
  • Subclassing would lead to an explosion of classes for every combination of features.
  • Examples: Java I/O streams (BufferedReader(new FileReader(...))), UI components with scrollbars/borders.
How it works
  1. Component interface: declares the operation that will be decorated.
  2. ConcreteComponent: the base object that can be wrapped.
  3. BaseDecorator: implements Component and holds a reference to a Component (the wrapped object).
  4. ConcreteDecorators: extend BaseDecorator, adding behaviour before/after calling the wrapped component.
  5. Stack decorators: new LoggingDecorator(new CachingDecorator(new RealService())).
UML Diagram
Java Code Example
// Component
interface Notifier { void send(String msg); }

// ConcreteComponent
class EmailNotifier implements Notifier {
    @Override public void send(String msg) { System.out.println("Email: " + msg); }
}

// BaseDecorator
abstract class NotifierDecorator implements Notifier {
    protected final Notifier wrapped;
    NotifierDecorator(Notifier n) { wrapped = n; }
    @Override public void send(String msg) { wrapped.send(msg); }
}

// ConcreteDecorator 1
class SMSDecorator extends NotifierDecorator {
    SMSDecorator(Notifier n) { super(n); }
    @Override public void send(String msg) {
        super.send(msg); System.out.println("SMS: " + msg);
    }
}

// ConcreteDecorator 2
class SlackDecorator extends NotifierDecorator {
    SlackDecorator(Notifier n) { super(n); }
    @Override public void send(String msg) {
        super.send(msg); System.out.println("Slack: " + msg);
    }
}

// Stack decorators at runtime!
Notifier n = new SlackDecorator(
                 new SMSDecorator(
                     new EmailNotifier()));
n.send("Server is down!");
// Email: Server is down!  SMS: Server is down!  Slack: Server is down!
Design Principles Applied
  • OCP Add new decorators without touching existing components
  • Composition over Inheritance Stacking wrappers beats a class hierarchy explosion
  • SRP Each decorator adds exactly one responsibility
Interview Questions
Q: How does Decorator differ from Inheritance for adding behaviour?
With inheritance you fix the combination at compile time — EmailSMSSlackNotifier as a subclass. With Decorator you combine at runtime. 3 decorator types → 7 possible combinations without writing a single new subclass.
Q: What real Java example uses the Decorator pattern?
java.io streams: new BufferedReader(new InputStreamReader(new FileInputStream("f"))). Each wrapper adds a capability (buffering, character encoding) without modifying the inner stream.
Q: What is the difference between Decorator and Composite?
Both use the same component interface. Decorator wraps exactly one component and adds behaviour to it. Composite wraps multiple components and aggregates their behaviour. Intent: Decorator = augment; Composite = group.
Pitfalls & When NOT to Use
  • Many small decorator classes can clutter the codebase — document the intended stacking order.
  • Order of decoration matters and can lead to subtle bugs (e.g., caching before or after logging).
  • Debugging a deep decorator stack is hard — the call trace spans many layers.
  • If only one combination ever exists, a single subclass is simpler.
08 Bridge Structural
What it is

The Bridge pattern decouples an abstraction from its implementation so that the two can vary independently. It splits a large class (or set of related classes) into two separate hierarchies — Abstraction and Implementation — connected by a composition reference ("bridge"). This avoids the Cartesian product explosion of subclasses.

When to use it
  • You want to avoid a permanent binding between an abstraction and its implementation so both can evolve independently.
  • You're facing a class hierarchy explosion due to multiple independent dimensions (e.g., Shape × Color, RemoteControl × Device).
  • You want to switch implementations at runtime without changing the abstraction's code.
  • Examples: device drivers, cross-platform rendering, remote control systems.
How it works
  1. Identify two independent dimensions (e.g., Shape and Drawing API).
  2. Create an Implementor interface for one dimension (e.g., DrawingAPI).
  3. Create the Abstraction class that holds a reference (the bridge) to the Implementor interface.
  4. Refine the Abstraction with concrete subclasses (e.g., Circle, Square) that call the implementor's methods.
  5. Provide ConcreteImplementors (e.g., SvgAPI, CanvasAPI) that can be swapped at runtime.
UML Diagram
Java Code Example
// Implementor interface
interface DrawingAPI {
    void drawCircle(double x, double y, double r);
}

// Concrete implementors
class SvgAPI implements DrawingAPI {
    @Override public void drawCircle(double x, double y, double r) {
        System.out.println("SVG circle at ("+x+","+y+") r="+r);
    }
}
class CanvasAPI implements DrawingAPI {
    @Override public void drawCircle(double x, double y, double r) {
        System.out.println("Canvas arc at ("+x+","+y+") r="+r);
    }
}

// Abstraction — holds bridge to DrawingAPI
abstract class Shape {
    protected DrawingAPI api; // THE BRIDGE
    Shape(DrawingAPI api) { this.api = api; }
    abstract void draw();
}

// Refined abstraction
class Circle extends Shape {
    double x, y, r;
    Circle(double x, double y, double r, DrawingAPI api) {
        super(api); this.x=x; this.y=y; this.r=r;
    }
    @Override void draw() { api.drawCircle(x, y, r); } // delegates
}

// Usage — swap implementation without changing Circle
Shape s1 = new Circle(5, 10, 3, new SvgAPI());
Shape s2 = new Circle(5, 10, 3, new CanvasAPI());
s1.draw(); // SVG circle...
s2.draw(); // Canvas arc...
Design Principles Applied
  • OCP Both abstraction and implementation dimensions open for extension
  • Composition over Inheritance Avoids the N×M subclass explosion
  • Program to Interface Abstraction depends on DrawingAPI, not SvgAPI
  • SRP Shape handles geometry; DrawingAPI handles rendering
Interview Questions
Q: What problem does Bridge solve that normal inheritance doesn't?
With inheritance, adding a new shape AND a new drawing API requires O(M×N) subclasses. Bridge splits these into two independent hierarchies so you only need O(M+N) classes — one set for shapes, one for APIs, connected by composition.
Q: How is Bridge different from Strategy?
They look similar — both use composition to delegate to an interface. Intent differs: Strategy replaces an algorithm at runtime (the context knows which strategy to apply). Bridge separates two orthogonal class hierarchies; the implementor is often set once at construction.
Q: When do you know you need Bridge instead of simple composition?
When you have two separate reasons to vary — two independent dimensions each with their own hierarchy (e.g., Shape and Color, RemoteControl and Device). If only one dimension varies, Strategy or simple composition is sufficient.
Pitfalls & When NOT to Use
  • Over-engineering risk: if only one implementation ever exists, Bridge adds indirection with no benefit.
  • The pattern adds complexity — two class hierarchies to understand instead of one.
  • Don't confuse with Adapter: Adapter fixes an existing mismatch; Bridge is planned upfront to enable independent variation.

09 Strategy Behavioral
What it is

The Strategy pattern defines a family of algorithms, encapsulates each one, and makes them interchangeable at runtime. The client can choose or switch algorithms without changing the class that uses them.

When to use it
  • A class needs many variants of an algorithm (e.g., sorting, payment methods, compression).
  • You want to eliminate large if/else or switch chains selecting algorithm variants.
  • You want to share behaviour across unrelated classes without inheritance.
How it works
  1. Create a Strategy interface with one method (the algorithm's entry point).
  2. Each algorithm is a concrete class implementing that interface.
  3. The Context class holds a reference to the Strategy interface.
  4. A setter (setStrategy()) lets the caller swap strategies at runtime.
  5. The context's action method delegates to the strategy object.
UML Diagram
Java Code Example
// Strategy interface
interface SortStrategy { void sort(int[] data); }

// Concrete strategies
class BubbleSort implements SortStrategy {
    @Override public void sort(int[] data) { System.out.println("Bubble sorting"); }
}
class QuickSort implements SortStrategy {
    @Override public void sort(int[] data) { System.out.println("Quick sorting"); }
}

// Context — delegates to the strategy
class Sorter {
    private SortStrategy strategy;
    public void setStrategy(SortStrategy s) { strategy = s; }
    public void sort(int[] data) { strategy.sort(data); }
}

// Usage — swap strategy at runtime
Sorter sorter = new Sorter();
sorter.setStrategy(new BubbleSort());
sorter.sort(new int[]{3,1,2}); // Bubble sort
sorter.setStrategy(new QuickSort());
sorter.sort(new int[]{3,1,2}); // Now quick sort
Design Principles Applied
  • Encapsulate What Varies Algorithms vary — pulled out into their own classes
  • Program to Interface Context depends on SortStrategy, not concrete classes
  • Composition over Inheritance Behaviour composed via has-a, not subclassing
  • OCP Add new strategies without touching the context
Interview Questions
Q: How is Strategy different from a simple if/else?
With if/else, adding a new algorithm requires modifying the existing class (violates OCP). With Strategy, you add a new class implementing the interface — zero change to the context. It also enables runtime switching and unit testing each algorithm in isolation.
Q: What's the relationship between Strategy and delegation?
Strategy uses delegation as its implementation mechanism. The context delegates the algorithm to the strategy object. Strategy is the pattern; delegation is the structural technique it relies on.
Pitfalls & When NOT to Use
  • Overkill if you only have two algorithms that will never change — a simple boolean flag may be cleaner.
  • Clients must be aware of different strategies to select the right one — increased coupling at the selection site.
  • Too many strategy classes can clutter the codebase if variants are minor.
10 Observer Behavioral
What it is

The Observer pattern defines a one-to-many dependency: when a subject object changes state, all its registered observers are notified automatically. Like a newsletter — the publisher sends to anyone who subscribed; no one else receives it.

When to use it
  • A change in one object requires changing others, and you don't know how many objects need to change.
  • Objects should be able to notify other objects without making assumptions about who those objects are.
  • Event-driven UIs, model-view synchronisation (MVC), message queues.
How it works
  1. Create a Subject interface with addObserver, removeObserver, notify.
  2. Create an Observer interface with an update() method.
  3. Concrete Subject stores a list of observers; calls update() on each when it changes.
  4. Concrete Observers implement update() to react to the notification.
UML Diagram
Java Code Example
import java.util.*;

interface Observer { void update(float temp); }

class WeatherStation {
    private List<Observer> observers = new ArrayList<>();
    private float temperature;

    public void addObserver(Observer o)    { observers.add(o); }
    public void removeObserver(Observer o) { observers.remove(o); }

    public void setTemperature(float t) {
        temperature = t;
        notifyObservers();
    }

    private void notifyObservers() {
        for (Observer o : observers) o.update(temperature);
    }
}

class PhoneDisplay implements Observer {
    @Override public void update(float temp) {
        System.out.println("Phone shows: " + temp + "°C");
    }
}

// Usage
WeatherStation station = new WeatherStation();
station.addObserver(new PhoneDisplay());
station.setTemperature(22.5f); // → Phone shows: 22.5°C
Design Principles Applied
  • OCP Add observers without modifying the subject
  • Loose Coupling Subject knows observers only via interface
  • Program to Interface Subject holds Observer references, not concrete types
Interview Questions
Q: What is the difference between push and pull models in Observer?
Push: the subject sends the new data in the update() call. Pull: the subject passes itself to update(Subject s) and observers query only what they need. Pull is more flexible; push is simpler but may send unnecessary data.
Q: How does Observer relate to the event-listener pattern in Java?
Java's ActionListener, EventListener, and reactive frameworks like RxJava are all implementations of the Observer pattern. The pattern is the foundation of all event-driven programming.
Pitfalls & When NOT to Use
  • Memory leaks: if observers are never removed, the subject holds references indefinitely (use WeakReference or always unsubscribe).
  • Unexpected update order: observer notification order is usually unspecified — don't rely on it.
  • Cascade updates: an observer changing the subject can cause infinite update loops.
11 State Behavioral
What it is

The State pattern allows an object to alter its behaviour when its internal state changes. The object appears to change its class. It is structurally identical to Strategy — but the intent is different: State manages state transitions, Strategy manages interchangeable algorithms.

When to use it
  • An object has many states and its behaviour differs radically in each state.
  • State-specific behaviour is currently implemented with large conditional blocks.
  • State transitions need to be explicit and managed (e.g., a vending machine, traffic light, order lifecycle).
How it works
  1. Create a State interface with methods representing behaviours.
  2. Create concrete state classes — each implements the interface differently.
  3. The Context class holds a reference to the current State.
  4. State transitions: either the context changes state, or the state objects change it themselves.
UML Diagram
Java Code Example
interface LightState { void change(TrafficLight ctx); }

class RedState implements LightState {
    @Override public void change(TrafficLight ctx) {
        System.out.println("Red → Green");
        ctx.setState(new GreenState());
    }
}
class GreenState implements LightState {
    @Override public void change(TrafficLight ctx) {
        System.out.println("Green → Yellow");
        ctx.setState(new YellowState());
    }
}
class YellowState implements LightState {
    @Override public void change(TrafficLight ctx) {
        System.out.println("Yellow → Red");
        ctx.setState(new RedState());
    }
}

class TrafficLight {
    private LightState state = new RedState();
    public void setState(LightState s) { state = s; }
    public void change() { state.change(this); }
}

TrafficLight light = new TrafficLight();
light.change(); // Red → Green
light.change(); // Green → Yellow
light.change(); // Yellow → Red
Design Principles Applied
  • Encapsulate What Varies State-specific behaviour encapsulated per state class
  • OCP Add new states without modifying the context
  • SRP Each state class handles one state's behaviour
Interview Questions
Q: What's the key difference between State and Strategy?
Intent. Strategy is about choosing an algorithm; the context usually doesn't care which strategy runs. State is about managing a lifecycle of transitions — states are aware of each other and trigger transitions. Clients choose strategies; state objects manage their own transitions.
Q: Who should manage state transitions — the Context or the State objects?
Either approach is valid. State-managed transitions keep transition logic decentralised but create coupling between state classes. Context-managed transitions centralise logic but grow the context class as states increase.
Pitfalls & When NOT to Use
  • With only two states, a simple boolean flag is cleaner.
  • State objects that know about each other create tight coupling — document transitions clearly.
  • State explosion: a complex state machine may need a dedicated FSM library rather than raw State pattern.
12 Chain of Responsibility Behavioral
What it is

The Chain of Responsibility (CoR) pattern passes a request along a chain of handlers. Each handler decides either to process the request or forward it to the next handler in the chain. Think of it as a linked list of decision-makers.

When to use it
  • More than one object may handle a request, and the handler isn't known a priori.
  • You want to issue a request to one of several handlers without specifying the receiver explicitly.
  • Examples: middleware pipelines, logging levels, customer support escalation (L1 → L2 → Manager).
How it works
  1. Define a Handler interface with handleRequest() and a reference to its successor.
  2. Each concrete handler tries to process the request. If it can't, it calls successor.handleRequest().
  3. Chain the handlers at setup time; the order defines processing priority.
UML Diagram
Java Code Example
abstract class SupportHandler {
    protected SupportHandler successor;
    public void setSuccessor(SupportHandler s) { successor = s; }
    public abstract void handleRequest(int level);
}

class L1Support extends SupportHandler {
    @Override public void handleRequest(int level) {
        if (level <= 1) System.out.println("L1 resolved");
        else if (successor != null) successor.handleRequest(level);
    }
}

class L2Support extends SupportHandler {
    @Override public void handleRequest(int level) {
        if (level <= 2) System.out.println("L2 resolved");
        else if (successor != null) successor.handleRequest(level);
        else System.out.println("Unresolved!");
    }
}

L1Support l1 = new L1Support();
L2Support l2 = new L2Support();
l1.setSuccessor(l2);
l1.handleRequest(1); // L1 resolved
l1.handleRequest(2); // L2 resolved
l1.handleRequest(3); // Unresolved!
Design Principles Applied
  • SRP Each handler handles one level of concern
  • OCP Insert new handlers in the chain without touching existing ones
  • Loose Coupling Sender doesn't know which handler will process the request
Interview Questions
Q: What data structure does CoR resemble?
A singly linked list. Each handler node has a reference to the next (successor), and the request travels down the chain until it's handled or the chain ends.
Q: What happens if no handler processes the request?
The request falls off the end of the chain unhandled. You should design the last handler to catch-all, log, or throw an exception. Never silently drop requests.
Pitfalls & When NOT to Use
  • Debugging is harder — it's non-obvious which handler ultimately processed the request.
  • Performance: in the worst case, all handlers are traversed for every request.
  • Don't use when only one handler will ever be relevant — direct call is simpler.
13 Composite Behavioral
What it is

The Composite pattern composes objects into tree structures to represent part-whole hierarchies. It lets clients treat individual objects (leaves) and compositions of objects (composites) uniformly through a common interface.

When to use it
  • You need to represent hierarchies: file systems, org charts, UI component trees, XML/HTML DOMs.
  • You want clients to ignore the difference between individual elements and groups.
How it works
  1. Component: the common interface for all elements (leaves and composites).
  2. Leaf: a basic element with no children. Implements Component.
  3. Composite: a container that holds a collection of children (Components). Implements Component by delegating to children.

Transparent vs Safe: Transparent — child management methods in Component (Leaf throws exceptions). Safe — child management only in Composite class (client must cast).

UML Diagram
Java Code Example
interface FileSystemItem { void display(String indent); }

// Leaf
class File implements FileSystemItem {
    private String name;
    File(String name) { this.name = name; }
    @Override public void display(String indent) {
        System.out.println(indent + "📄 " + name);
    }
}

// Composite
class Folder implements FileSystemItem {
    private String name;
    private List<FileSystemItem> children = new ArrayList<>();
    Folder(String name) { this.name = name; }
    public void add(FileSystemItem item) { children.add(item); }
    @Override public void display(String indent) {
        System.out.println(indent + "📁 " + name);
        for (FileSystemItem child : children)
            child.display(indent + "  "); // recursive
    }
}

Folder root = new Folder("root");
root.add(new File("readme.txt"));
Folder src = new Folder("src");
src.add(new File("Main.java"));
root.add(src);
root.display(""); // uniform call — same for file and folder
Design Principles Applied
  • Program to Interface Client uses Component, not concrete Leaf/Composite
  • OCP Add new leaf/composite types without changing client code
  • DRY Recursive doThis() avoids traversal logic duplication
Interview Questions
Q: What is the tradeoff between Transparent and Safe Composite?
Transparent: add/remove in Component. All clients can call them uniformly, but Leaf must throw UnsupportedOperationException — breaks safety. Safe: add/remove only in Composite. Type-safe, but clients must cast, losing uniformity.
Q: When does Composite use recursion?
When the Composite's operation iterates its children and calls the same operation on each child, which may itself be a Composite — creating a recursive tree traversal.
Pitfalls & When NOT to Use
  • Difficult to restrict the types of children — any Component subtype can be added.
  • Overly general design can make it harder to restrict domain-specific rules.
  • Not appropriate for flat, non-hierarchical data — adds unnecessary indirection.
SOLID Principles Design Principles
S Single Responsibility Principle SOLID
What it is

The Single Responsibility Principle states that a class should have only one reason to change. In practice this means every class, module, or function should be responsible for exactly one slice of the system's behaviour. "Reason to change" is the key phrase — if two different stakeholders (e.g., the billing team and the reporting team) could independently ask you to modify the same class, it has too many responsibilities.

When to apply it
  • A class grows beyond ~200 lines and starts feeling like it "does everything".
  • You find yourself changing a class for unrelated reasons across sprints.
  • Writing unit tests forces you to mock unrelated dependencies (sign of mixed concerns).
  • The class name uses an "and" or "or" — e.g., UserParserAndEmailSender.
How it works
  1. Identify the distinct actors (stakeholders / roles) who could request a change to the class.
  2. Group functionality that serves the same actor into the same class.
  3. Extract functionality that serves a different actor into a new dedicated class.
  4. Wire the classes together via composition or dependency injection.
UML Diagram
Java Code Example
❌ Violation
// BAD: one class handles payroll, persistence, AND reporting
class Employee {
    public void calculatePay()    { /* CFO cares about this */ }
    public void saveToDatabase()  { /* DBA cares about this */ }
    public String generateReport() { /* COO cares about this */ }
    // Any of three teams can force a change here — too many reasons!
}
✅ Correct
// GOOD: each class has exactly one actor and one reason to change
class PayCalculator     { public void calculatePay(Employee e)  { /* CFO */ } }
class EmployeeRepository { public void save(Employee e)         { /* DBA */ } }
class ReportGenerator   { public String generate(Employee e)  { /* COO */ } }

// Employee itself only holds data
class Employee {
    private String name;
    private double salary;
    // getters/setters only
}
Interview Questions
Q: Isn't SRP just about keeping classes small?
No — size is a side-effect, not the goal. A class can be small and still violate SRP. The principle is about cohesion of purpose: all methods in the class serve the same actor. Uncle Bob defines "reason to change" specifically as a person or group who could request a change.
Q: How do you identify the boundaries of a "responsibility"?
Ask: "Who would ask me to change this method?" If the answer involves different teams or roles (CFO vs. CTO vs. operations), those are different responsibilities and should be in separate classes.
Q: Can SRP apply to methods, not just classes?
Yes — at the method level, SRP means a method does one thing at one level of abstraction. A method named processOrderAndSendEmail() is doing two things and should be split.
Pitfalls & Common Misconceptions
  • Over-splitting: Extracting every three-line block into its own class creates an explosion of tiny classes that are hard to navigate. Responsibility boundaries should reflect real-world actors, not code length.
  • Confusing SRP with DRY: SRP is about change ownership; DRY is about duplication. They are complementary but not the same.
  • Anemic domain model: Taking SRP to the extreme can result in classes with only data and no behaviour — pushing all logic into service classes, which can itself become a violation.
O Open/Closed Principle SOLID
What it is

The Open/Closed Principle states that software entities should be open for extension but closed for modification. Once a class is written and tested, adding new behaviour should be done by adding new code (a new class, a new implementation), not by editing the existing class. This protects stable, tested code from regressions while still allowing the system to grow.

When to apply it
  • You keep adding if/else or switch blocks to handle new types — each addition risks breaking existing logic.
  • A method that works perfectly today will need to handle additional variants in future sprints.
  • You want to ship a library or framework that others can extend without forking.
  • You have a stable core algorithm but need to vary a step (Strategy, Template Method).
How it works
  1. Identify the axis of variation — the dimension along which requirements will grow.
  2. Extract that variation behind an abstraction (interface or abstract class).
  3. The consuming class depends on the abstraction, not the concrete type.
  4. New behaviour is added by implementing the abstraction in a new class — the original class is never touched.
UML Diagram
Java Code Example
❌ Violation
// BAD: every new discount type requires editing this method
class DiscountCalculator {
    public double calculate(String type, double price) {
        if (type.equals("seasonal")) return price * 0.8;
        if (type.equals("loyalty"))  return price * 0.9;
        // adding "flash" discount means touching this class again ↑
        return price;
    }
}
✅ Correct
// GOOD: new discounts are added by creating new classes only
interface DiscountStrategy { double apply(double price); }

class SeasonalDiscount implements DiscountStrategy {
    public double apply(double p) { return p * 0.8; }
}
class LoyaltyDiscount implements DiscountStrategy {
    public double apply(double p) { return p * 0.9; }
}
// Adding flash sale: just add FlashDiscount class — zero changes above
class FlashDiscount implements DiscountStrategy {
    public double apply(double p) { return p * 0.5; }
}

class DiscountCalculator {
    public double calculate(DiscountStrategy strategy, double price) {
        return strategy.apply(price); // closed for modification ✓
    }
}
Interview Questions
Q: What design patterns directly embody OCP?
Strategy (vary algorithm), Template Method (vary a step), Decorator (add behaviour), and Observer (add observers without changing the subject) all achieve OCP by allowing extension via new classes rather than modification of existing ones.
Q: "Closed for modification" — does that mean I can never fix bugs in a class?
Bug fixes are allowed. "Closed for modification" means closed to new feature additions that alter the class's original contract. The principle targets scope creep inside stable components, not corrections to existing behaviour.
Q: When is OCP NOT worth applying?
When you cannot predict the axis of variation, abstracting prematurely creates the wrong interface. The principle is most valuable when a point of variation is already proven — do not apply it speculatively. Martin Fowler: "Three strikes and you refactor."
Pitfalls & Common Misconceptions
  • Premature abstraction: Wrapping everything in interfaces "just in case" adds indirection for no benefit. Apply OCP after you've seen a change happen once, not before.
  • Confusing with immutability: OCP doesn't mean classes are final or frozen — it means adding behaviour through new types, not editing existing logic.
  • Plugin architecture overkill: Not every if-statement needs to become a Strategy. Evaluate whether the variation is truly a business axis or just a minor detail.
L Liskov Substitution Principle SOLID
What it is

The Liskov Substitution Principle states that objects of a subtype must be substitutable for objects of their supertype without altering the correctness of the program. In everyday terms: if your code works with a Bird, it should work equally well with any Bird subclass — no surprises, no exceptions thrown, no silent behaviour changes. The subclass must honour the parent's contract, not just its method signatures.

When to apply it
  • A subclass overrides a parent method by throwing UnsupportedOperationException — classic LSP red flag.
  • Client code has instanceof checks to handle a specific subclass differently.
  • Inheriting from a class only to reuse some methods, not because the subclass truly is-a parent type.
  • A subclass weakens preconditions or strengthens postconditions of inherited methods.
How it works
  1. Verify the subclass satisfies the behavioural contract of the parent, not just the type signature.
  2. Subtype preconditions must be equal or weaker than the parent (accept at least what the parent accepts).
  3. Subtype postconditions must be equal or stronger than the parent (guarantee at least what the parent guarantees).
  4. If a subclass cannot honour the full contract, prefer composition over inheritance.
UML Diagram
Java Code Example
❌ Violation
class Bird { public void fly() { System.out.println("Flap wings"); } }

// Ostrich cannot fly — throws at runtime, breaking substitution
class Ostrich extends Bird {
    @Override
    public void fly() { throw new UnsupportedOperationException("Can't fly"); }
}

// Client code explodes when passed an Ostrich — LSP violated
void makeFly(Bird b) { b.fly(); }
✅ Correct
// GOOD: split the hierarchy so contracts are honoured
abstract class Bird       { public abstract void eat(); }
abstract class FlyingBird extends Bird { public abstract void fly(); }

class Sparrow extends FlyingBird {
    public void fly() { System.out.println("Sparrow flies"); }
    public void eat() { System.out.println("Sparrow eats"); }
}
// Ostrich extends Bird but NOT FlyingBird — no broken promises
class Ostrich extends Bird {
    public void eat() { System.out.println("Ostrich eats"); }
}

// Only FlyingBirds are passed where flying is expected
void makeFly(FlyingBird b) { b.fly(); }
Interview Questions
Q: What is the classic LSP violation example, and how do you fix it?
Square extending Rectangle — setting width also changes height in a Square, breaking the Rectangle contract. Fix: don't use inheritance. Either both implement a Shape interface without shared setters, or use composition.
Q: How does LSP relate to polymorphism?
LSP defines the conditions under which polymorphism is safe. Polymorphism lets you treat objects uniformly; LSP guarantees the substitution won't cause surprises. Without LSP, polymorphic code is unreliable.
Q: What does "contract" mean in LSP?
A contract includes: preconditions (what the method requires on input), postconditions (what it promises on output), and invariants (properties that hold throughout the object's lifetime). A subclass must not violate any of these.
Pitfalls & Common Misconceptions
  • "Is-a" in English ≠ is-a in code: An ostrich is a bird biologically, but it shouldn't extend a FlyingBird. Type hierarchies must reflect behavioural contracts, not real-world taxonomies.
  • Throwing exceptions in overrides: Overriding a method only to throw is always an LSP violation — it weakens postconditions to nothing.
  • Empty overrides: A subclass silently doing nothing in an override (no-op) can also violate LSP if the parent's contract promises a side effect.
I Interface Segregation Principle SOLID
What it is

The Interface Segregation Principle states that clients should not be forced to depend on methods they do not use. A fat interface that bundles many unrelated methods forces implementing classes to provide stubs or throw exceptions for methods they don't need. The fix is to break the fat interface into smaller, focused interfaces so each client only sees what's relevant.

When to apply it
  • A class implements an interface but leaves some methods as empty stubs or throws UnsupportedOperationException.
  • Adding a method to an interface forces changes in every single implementing class.
  • Two implementing classes share an interface but have completely different use cases.
  • A client imports a dependency only to use 1 of 10 methods on an interface.
How it works
  1. Identify distinct client roles that consume the interface.
  2. Group methods by role into smaller, cohesive interfaces — one per client type.
  3. Classes implement only the interfaces relevant to their role.
  4. If a class naturally fulfils multiple roles, it can implement multiple small interfaces.
UML Diagram
Java Code Example
❌ Violation
// BAD: RobotWorker is forced to implement eat() and sleep()
interface Worker {
    void work(); void eat(); void sleep(); void recharge();
}

class RobotWorker implements Worker {
    public void work()     { /* OK */ }
    public void recharge() { /* OK */ }
    public void eat()      { throw new UnsupportedOperationException(); } // ❌
    public void sleep()    { throw new UnsupportedOperationException(); } // ❌
}
✅ Correct
// GOOD: segregated interfaces; each class implements only what it needs
interface Workable    { void work(); }
interface Eatable     { void eat(); void sleep(); }
interface Rechargeable { void recharge(); }

// Human implements what makes sense for humans
class HumanWorker implements Workable, Eatable {
    public void work()  { System.out.println("Human works"); }
    public void eat()   { System.out.println("Human eats"); }
    public void sleep() { System.out.println("Human sleeps"); }
}
// Robot implements only Workable and Rechargeable — no forced stubs
class RobotWorker implements Workable, Rechargeable {
    public void work()     { System.out.println("Robot works"); }
    public void recharge() { System.out.println("Robot recharges"); }
}
Interview Questions
Q: What's the difference between ISP and SRP?
SRP is about a class having one reason to change. ISP is about an interface not forcing clients to depend on methods they don't use. They are complementary — SRP applied to interfaces produces ISP-compliant designs, but ISP specifically targets the interface consumers' perspective.
Q: How fine-grained should interfaces be?
Fine enough that each client role sees only what it needs — not so fine that you have one-method interfaces for everything. A good heuristic: if two clients always use the same subset of methods together, they belong in the same interface.
Q: How does Java 8's default methods relate to ISP?
Default methods let you add behaviour to an interface without breaking existing implementations, but they do not solve ISP — a fat interface with defaults still forces clients to compile against methods they don't use. ISP is about segregation, not about providing defaults.
Pitfalls & Common Misconceptions
  • One method per interface (over-segregation): Taking ISP to its extreme means every method gets its own interface — this creates an unmaintainable web of types. Group methods by the client role that uses them together.
  • Merging unrelated clients: Designing an interface around what's convenient to implement (one class that happens to do both) instead of what clients actually need leads back to fat interfaces.
  • Ignoring the client perspective: ISP is evaluated from the consumer's side. If all consumers use all methods, the interface is already well-segregated regardless of its size.
D Dependency Inversion Principle SOLID
What it is

The Dependency Inversion Principle has two rules: (1) High-level modules should not depend on low-level modules — both should depend on abstractions. (2) Abstractions should not depend on details; details should depend on abstractions. In practice: instead of a business class creating or importing a specific implementation (like a MySQL repository), it receives an interface at construction time. The wiring happens outside both classes.

When to apply it
  • Your business logic class directly instantiates a data access, notification, or infrastructure class with new.
  • Swapping a database, messaging queue, or external service requires changes deep inside domain classes.
  • Unit testing a class requires a real database or HTTP connection.
  • You want to introduce a new implementation of a service without touching the caller.
How it works
  1. Define an abstraction (interface) that describes what the high-level module needs, not what the low-level module provides.
  2. The high-level module accepts the abstraction via its constructor (preferred) or a setter.
  3. The low-level module implements the abstraction.
  4. A composition root (e.g., Spring context, main(), test setup) wires the concrete implementation to the abstraction at startup.
UML Diagram
Java Code Example
❌ Violation
// BAD: high-level OrderService directly creates a low-level MySQLOrderRepo
class OrderService {
    private MySQLOrderRepo repo = new MySQLOrderRepo(); // concrete dependency!
    public void placeOrder(Order o) { repo.save(o); }
    // Testing requires a real MySQL DB — and swapping to Postgres changes this class
}
✅ Correct
// GOOD: both high- and low-level depend on the abstraction
interface IOrderRepository { void save(Order order); }

class MySQLOrderRepository implements IOrderRepository {
    public void save(Order o) { System.out.println("Saving to MySQL: " + o); }
}

// For tests — swap without touching OrderService
class InMemoryOrderRepository implements IOrderRepository {
    public void save(Order o) { System.out.println("Saving in memory: " + o); }
}

// High-level module receives abstraction via constructor injection
class OrderService {
    private final IOrderRepository repo;
    OrderService(IOrderRepository repo) { this.repo = repo; } // DI
    public void placeOrder(Order o) { repo.save(o); }
}

// Composition root wires it together
OrderService svc = new OrderService(new MySQLOrderRepository());
// Test: new OrderService(new InMemoryOrderRepository())
Interview Questions
Q: What's the difference between DIP, Dependency Injection, and IoC?
DIP is the principle: depend on abstractions. Dependency Injection (DI) is a technique: pass dependencies from outside rather than creating them internally. Inversion of Control (IoC) is the broader pattern: a framework calls your code rather than the reverse. DI is one implementation of IoC, and DI is a common way to achieve DIP.
Q: Who owns the abstraction in DIP — the high-level or low-level module?
The high-level module owns (defines) the interface, because it expresses what it needs. The low-level module then implements that interface. This is the "inversion" — normally the high-level module would import and depend on the low-level; DIP flips this so both point at the abstraction.
Q: Can DIP be achieved without a DI framework like Spring?
Yes — constructor injection in plain Java is sufficient. Pass the dependency as a constructor parameter and wire everything in main(). Spring, Guice, and CDI automate this wiring but the principle itself requires no framework.
Pitfalls & Common Misconceptions
  • DIP ≠ "always use an interface": Wrapping every class in an interface for its own sake adds noise. Apply DIP where the dependency is a genuine variability point (persistence, messaging, external services) — not for stable domain value objects.
  • Injection via setters vs constructors: Constructor injection makes dependencies explicit and the object always valid. Setter injection allows optional dependencies but can leave an object in an invalid state if the setter is never called.
  • Service Locator anti-pattern: Pulling dependencies from a global registry (like ServiceLocator.get(IRepo.class)) appears to follow DIP but hides dependencies — prefer explicit constructor injection.
